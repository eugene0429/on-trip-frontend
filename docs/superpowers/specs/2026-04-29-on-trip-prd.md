## 1. 한 줄 요약

> 여행지에서 같은 동네에 와있는 다른 일행을 만나는 GPS 기반 매칭 앱. 지도에 여행자 인구가 보이고, 마음에 드는 일행을 콕 찔러 서로 찌르면 자동 매칭 → 1:1 채팅으로 이어진다.

## 2. 배경 · 문제 · 목표

### 2.1 사용자 문제
- 여행지에서 친구들과 시간을 보내다 보면 "다른 일행과 합류하면 더 재미있겠다"는 순간이 자주 생긴다 (술자리·맛집·액티비티).
- 그러나 현장에서 즉흥적으로 다른 일행에게 말 거는 건 사회적 부담이 크다.
- 기존 매칭 앱은 "1:1 데이팅" 색채가 강해 친구들과 같이 노는 컨셉과 맞지 않는다.
- 여행 커뮤니티(트립닷컴/마이리얼트립 등)는 정보 공유에 가깝고, 즉시·실시간 매칭이 어렵다.

### 2.2 프로덕트 목표
- **즉시성:** 지금 같은 거점에 있는 다른 일행을 1분 안에 발견할 수 있게 한다.
- **그룹 친화성:** 1:1 데이팅 색채를 누르고, 일행 단위 정보로 안전감을 준다.
- **풋풋함:** 키치 스티커 무드로 데이팅 앱과 시각적으로 차별화하고 진입 부담을 낮춘다.

### 2.3 비즈니스 목표
- **MVP:** 6개월 내 누적 가입자 30,000명 / 일 매칭 성사 100건 / D7 리텐션 25% 이상.
- **V2 출시 시점(IAP 도입):** 결제전환율 5%, ARPPU ₩15,000 목표.

## 3. 타깃 사용자

- **1차:** 20·30대 한국 국내 여행자 (커플·친구 그룹·동호회·솔로)
- **여성 사용자 비중**을 일찍 30% 이상 확보하는 것이 핵심 KPI (매칭 앱 생태계 활성화 조건).
- **지역:** 한국 전 지역 (강릉·부산·경주·제주·서울 등 인기 거점에 사용자 집중 예상)

| 항목 | 결정 |
|---|---|
| 매칭 단위 | 그룹 ↔ 그룹 (솔로/그룹 모두 자유 등록·매칭) |
| 서비스 범위 | 한국 국내 한정 |
| 플랫폼 | iOS + Android (React Native + Expo) |

| 채팅 | 1:1 (대표자 ↔ 대표자) — V2에서 단톡 확장 |

| UI/UX 톤 | 키치 스티커 (두꺼운 검은 라인 + 비비드) |

## 5. 사용자 플로우

### 5.1 첫 사용 (가입)

```
앱 설치 → 온보딩 (3 슬라이드)
  → PASS 본인인증 (휴대폰/이름/생년월일/성별 확인)
  → 닉네임 + 한줄소개 + MBTI(선택) + 프로필 사진(선택) 입력
  → 위치 권한 / 알림 권한 동의
  → 가입 완료 → 1,000코인 자동 지급 안내
  → 지도 메인 진입
```

### 5.2 메인 사용 루프

```
지도 진입
  ├─ 광역 줌(시·도): "부산 1,234명" 등 행정구역 인구
  ├─ 세부 줌(거점): "광안리 24명", "해운대 41명" 핀
  └─ 거점 핀 탭 → 거점 상세

거점 상세
  └─ 등록된 일행 카드 리스트 (대표자 사진+닉네임+일행 요약)
       └─ 카드 탭 → 일행 프로필 상세
              └─ 콕 찌르기(100코인) + 한마디(50자)

콕 → 매칭
  ├─ 우연히 서로 콕: 즉시 매칭 + 채팅방 생성
  └─ 한쪽이 먼저 콕:
       상대방의 [받은 콕] 탭에 추가
        → 상대가 [프로필 열람](500코인) → 누가 보낸지 확인
        → 상대가 맞콕(100코인) → 매칭 + 채팅방

채팅
  └─ 1:1 무료 무제한 / 자동 마스킹 / 신고·차단 가능
```

### 5.3 등록 (여행왔어요)

```
지도 메인 하단 [여행 왔어요] CTA
  → GPS 읽음 → 가까운 거점 자동 제안 (수동 변경 가능)
  → 일행 정보 입력:
       인원수 (1명 / 2명 / 3명 / 4명+)
       성별 구성 (예: 남2 여1)
       평균 나이대 (20초/20후/30초/30후/40+)
       일행 한줄소개 (50자)
       여행 컨셉 다중선택 (맛집 / 술자리 / 액티비티 / 관광 / 사진)
       여행 기간 (당일치기 / 1박2일 / 2박3일 / 3박4일 / 4박+)
  → 등록 완료 → 같은 거점에 노출
```

### 5.4 등록 만료 정책

- 등록 시 선택한 여행 기간 종료일에 자동 비공개 (시각 = 종료일 23:59)
- 매일 아침 9시 푸시: "오늘도 ○○에 계신가요? (예 / 일정 종료)"
- 마지막 활동(앱 열기/메시지/푸시 응답) **후 24시간 무반응 시 자동 비공개** (스팸 정리)
- 만료 후에도 등록 이력은 보관 / 매칭·채팅방 유지 (만료 = 신규 노출 중단)
- 일정 도중 다른 지역 이동 시 [현재 지역 변경] → 이전 등록 종료 + 새 등록

## 8. 비즈니스 모델 상세

### 8.1 코인 차감 정책

| 액션 | 코인 | 비고 |
|---|---|---|
| 가입 보너스 | +1,000 | 1회성 |
| 콕 찌르기 | −100 | 매번 차감, 매일 리프레시 X |
| 받은 콕 프로필 열람 | −500 | 동일 대상 재열람 무료 |
| 매칭(자동) | 0 | 양쪽 콕 비용으로 이미 차감됨 |
| 채팅 메시지 | 0 | 무제한 무료 |
| 신고로 인한 환불 | +N | 운영자 승인 시 (어뷰저 콕 환불) |

### 8.2 매칭 메커니즘

1. **케이스 A — 동시 콕 (우연):**
   `User1 → Trip2`로 콕 + `User2 → Trip1`으로 콕이 모두 존재 → 즉시 Match 생성, 양쪽 채팅방 알림.
2. **케이스 B — 비대칭 콕:**
   `User1 → Trip2`만 콕 → User2의 [받은 콕] 목록에 가려진 상태로 추가 → User2가 프로필 열람(500코인) → User2 → Trip1에 콕(100코인) → Match 생성.
3. **만료:** 콕은 sender 또는 receiver 어느 한쪽 Trip이 만료되면 노출 중단되지만 이력은 보관 (재방문 시 재활성 정책은 V2).

### 8.3 IAP (V2 도입 예정)

- MVP는 **무료 1,000코인만 운영**.
- 다 쓴 사용자에게는 "충전 곧 오픈! 알림 신청" 화면 노출.
- V2 패키지(가안): 1,000 ₩4,900 / 5,000 ₩19,900 / 12,000 ₩39,900 / 30,000 ₩89,900

## 9. 기능 요구사항 (FR)

### 9.1 인증/계정
- FR-001: PASS 본인인증을 통과해야만 가입 완료. 동일 phone_hash로 1계정만 허용.
- FR-002: 닉네임은 시스템 전체 UNIQUE, 2~12자, 한글/영문/숫자.
- FR-003: 탈퇴 시 개인정보는 30일 보관 후 영구 삭제. 탈퇴 30일 내 재가입은 차단.

### 9.2 지도/지역
- FR-010: 지도는 카카오맵 SDK 사용. 줌 레벨에 따라 광역(행정구역) ↔ 세부(POI) 자동 전환.
- FR-011: POI는 초기 큐레이션 200~300개. 사용자 GPS 기준 가장 가까운 POI까지 거리 표시.
- FR-012: GPS 기준 가장 가까운 POI까지 1km 초과 시 "근처 거점 없음" + 수동 선택 옵션 제공.
- FR-013: 인구수는 60초 캐시 (Redis), 새로고침 시 강제 갱신.

### 9.3 등록
- FR-020: 한 사용자는 동시에 1개의 활성 Trip만 보유.
- FR-021: Trip 정보는 등록 후 24시간 이내 1회 수정 가능.
- FR-022: 매일 09:00 푸시로 활성 사용자에게 일정 확인 요청.
- FR-023: 마지막 활동 24시간 무반응 시 자동 비공개 (status=expired).

### 9.4 콕/매칭
- FR-030: 코인 부족 시 콕 버튼 비활성화 + "충전 곧 오픈" 안내.
- FR-031: 동일 sender → receiver_trip 중복 콕 차단 (최초 1회만). receiver가 새 Trip을 등록하면 새 콕 가능.
- FR-032: 차단 관계는 양방향 콕 차단.
- FR-033: 매칭 성사 시 양쪽에 푸시 알림 + [활동] 뱃지.
- FR-034: 받은 콕 카드의 미리보기에서는 발신자 프로필을 가린 상태로 (1) 발신자 일행 정보(인원·성별구성·나이대·컨셉) 일부와 (2) 콕 한마디의 첫 10자 + "..."를 노출. 발신자 정체(닉네임/사진/한줄소개)는 프로필 열람(500코인) 시에만 공개.
- FR-035: 콕은 sender 또는 receiver Trip 중 한쪽이 만료되면 받은 콕 목록의 신규 노출 중단. 매칭이 이미 성사된 콕은 채팅방 유지를 위해 이력 보관.

### 9.5 채팅
- FR-040: 1:1 채팅, 매칭 즉시 채팅방 생성. 양쪽 Trip이 모두 만료되어도 채팅방은 유지.
- FR-041: 메시지 발송 시 다음 패턴 자동 마스킹 (콕 한마디에도 동일 적용):
  - 휴대폰 번호 (010-XXXX-XXXX 등 변형)
  - 카카오톡/인스타그램/라인 ID 의심 키워드
  - 계좌번호 (은행명 + 숫자)
- FR-042: 마스킹된 메시지는 발신자 화면에는 원본, 수신자 화면에는 "***" 표시. 발신자에게는 송신 직후 "외부 연락처는 가려져 전달돼요" 토스트 1회 표시.
- FR-043: 채팅방에서 [신고] [차단] 버튼 상시 노출. 차단 시 채팅방 즉시 종료.

### 9.6 안전/모더레이션
- FR-050: 신고 5종 카테고리 (스팸/사기, 부적절한 메시지, 욕설/모욕, 신체 노출, 기타).
- FR-051: 같은 사용자가 누적 신고 3회 → 자동 7일 정지, 5회 → 영구 정지.
- FR-052: 정지된 사용자의 phone_hash + device_id 기록 → 재가입 시 1차 거름.

## 10. 비기능 요구사항 (NFR)

- **NFR-001 (성능):** 지도 인구 카운트 응답 ≤ 300ms (p95).
- **NFR-002 (성능):** 콕 발사 → 매칭 처리 ≤ 1s.
- **NFR-003 (실시간):** 채팅 메시지 전달 지연 ≤ 500ms (p95).
- **NFR-004 (가용성):** 99.5% (MVP 기준).
- **NFR-005 (보안):** 모든 API HTTPS, JWT 인증, refresh 토큰.
- **NFR-006 (개인정보):** PASS로 받은 실명/생년월일은 암호화 저장, 다른 사용자에게 절대 노출 X.
- **NFR-007 (확장성):** 동접 1,000명 + DAU 10,000 처리 가능한 단일 인스턴스 + RDS + Redis.

### 15.5 협업 규칙

- **Git 전략:** `main` (배포), `develop` (통합), `feature/<scope>` (작업). PR은 develop으로 머지.
- **API 변경:** 백엔드가 OpenAPI 스펙 PR을 먼저 올리고, FE 리뷰 후 머지. 머지된 스펙 = 진실 (구현이 스펙을 따라감).
- **이슈 트래커:** Linear 또는 GitHub Projects (1순위). 모든 작업은 이슈로 생성, PR 본문에 이슈 번호 명시.
- **코드 리뷰:** 모든 PR은 상대방 1인 승인 필요. 24시간 내 리뷰 SLA.
- **로컬 환경:** Docker Compose로 PostgreSQL + Redis 동시 기동 스크립트 (BE 책임 제공).
- **시크릿 관리:** `.env.example` 커밋, 실제 키는 1Password/Doppler 공유.
- **에러 모니터링:** Sentry 양쪽 통합 (FE 앱 / BE 서버). 에러 발생 시 슬랙 알림.

## 17. 리스크 / 가정

### 리스크
1. **콜드 스타트:** 사용자 밀도 부족으로 매칭이 안 되면 즉시 이탈 → 초기 거점은 부산/강릉/제주 3곳에 집중 마케팅으로 시드.
2. **여성 사용자 확보 실패:** 1차 출시 직후 여성 비중 모니터링 필요. 30% 미만 시 신뢰 기능(B의 자동 마스킹·신고)을 마케팅 메시지로 강조.
3. **Apple 매칭 앱 심사:** 한국 매칭 앱은 본인인증/사진 정책에서 자주 리젝됨. PASS 적용으로 대응 가능하지만 심사 일정 +1~2주 가산.
4. **PASS 비용:** 인증 1회당 약 ₩50~150. 가입 트래픽 폭증 시 단기 비용 상승 → CAC 추적 필요.
5. **어뷰저 유입:** 무료 1,000코인 = 콕 10회로 어뷰저 매력적. 가입 직후 1시간 내 다수 콕 발사 패턴은 자동 플래그 + 운영자 검토.

### 가정
- 카카오맵 RN SDK 안정 버전이 사용 가능하다 (대안: 네이버 지도, 구글 지도).
- 10주 내 PASS 연동 + Apple/Google 심사 통과가 가능하다.
- 초기 사용자 200~300개 POI에 자연스럽게 분산된다.
